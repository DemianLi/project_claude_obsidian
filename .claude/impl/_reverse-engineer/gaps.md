# Phase 4：層次三 — 邏輯缺口識別

程式碼中**不完整、矛盾、或有疑問**的地方。
**這些缺口在釐清前，任何擴充功能都不應該依賴它們。**

## 缺口類型

| 類型 | 說明 |
|------|------|
| 🔘 孤立程式碼 | 有 function 定義但從未被呼叫 |
| ⚡ 狀態機缺口 | 狀態轉換有漏掉的路徑 |
| 🗃️ 資料孤島 | 有資料但沒有程式碼使用，或反過來 |
| 🔀 矛盾邏輯 | 兩個地方對同一件事有不同處理 |
| 🔢 魔術數字 | hardcode 的值不知道代表什麼 |

## 缺口格式

```
GAP-{N}：{缺口描述}
  類型：{⚡/🗃️/🔀/🔘/🔢}
  現象：{程式碼中觀察到的不完整/矛盾情形}
  影響：{擴充哪些功能前必須先釐清}
  建議問甲方：「{具體問題文字}」
```

範例：
```
GAP-001：status = 'processing' 的訂單無法到達 'shipped'
  類型：⚡ 狀態機缺口
  現象：狀態轉換：pending → processing → ? → shipped
         找不到 processing → shipped 的觸發條件
  影響：擴充出貨功能前必須先釐清
  建議問甲方：「訂單從『處理中』到『已出貨』的觸發條件是什麼？
               是人工操作還是系統自動？」

GAP-002：orders 表有 discount_code 欄位，但無 discount_codes 表
  類型：🗃️ 資料孤島
  影響：若新功能涉及折扣，需先了解此是廢棄還是未完成功能
```
