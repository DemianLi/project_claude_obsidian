# Phase 3：層次二 — 推測業務邏輯

從程式碼**推斷**業務邏輯，每條推測必須標記：
- 推測依據（從哪段程式碼推斷）
- 信心度（高 / 中 / 低）
- 需要驗證的問題

## 信心度定義

| 信心度 | 定義 | 範例 |
|--------|------|------|
| 🟢 高 | 程式碼邏輯非常明確，幾乎不可能有其他解釋 | `if status == 'shipped': send_email()` |
| 🟡 中 | 有合理推測依據，但存在其他可能的解釋 | 欄位命名暗示某種流程，但沒有完整實作 |
| 🔴 低 | 高度不確定，可能是廢棄程式碼或特例 | 死程式碼、被 comment 的邏輯、魔術數字 |

## 推測格式

```
INF-{N}：{業務邏輯描述}
  推測依據：{1-2 行關鍵程式碼引用}
  信心度：🟢 高 / 🟡 中 / 🔴 低
  需驗證：{需要問甲方的問題}
```

範例：
```
INF-001：訂單只能由系統取消，不能由用戶取消
  推測依據：/api/orders/{id}/cancel 路由不存在；
            cancel 狀態只出現在後台管理 controller
  信心度：🟡 中
  需驗證：是否曾有取消功能後來被移除？

INF-002：出貨後不能修改訂單
  推測依據：status = 'shipped' 後，update controller 會 return 403
  信心度：🟢 高
  需驗證：有無例外情況（例如數量錯誤的修正）？
```
