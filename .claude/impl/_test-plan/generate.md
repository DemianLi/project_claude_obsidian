# Phase 2：為每條需求生成測試情境

針對每條 ✅ 已確認需求，產生三類測試案例：

## 情境格式

```markdown
#### TC-{需求ID}-{序號}：{測試情境標題}

**類型**：正向測試 / 負向測試 / 邊界測試
**對應需求**：{REQ-ID} {需求名稱}
**前置條件**：{系統/資料的初始狀態}
**測試步驟**：
  1. {步驟一}
  2. {步驟二}
**預期結果**：{系統應有的反應，與驗收條件對應}
**測試狀態**：⬜ 未測
**備註**：{環境要求、特殊注意事項}
```

## 三類測試案例生成規則

**1. 正向測試（Happy Path）**
- 每條驗收條件至少 1 個正向測試
- 使用最典型的正常使用情境
- 步驟清晰到讓不熟悉系統的人也能執行

**2. 負向測試（Negative Cases）**
必填欄位為空 / 格式錯誤（email、日期）/ 超出範圍（負數、最大值）
/ 無權限的操作 / 重複操作（重複送出、重複建立）

**3. 邊界測試（Edge Cases）**
最小值/最大值 / 空列表/大量資料 / 同時操作（並發）
/ 中途中斷（網路斷線、頁面重整）/ 特殊字元（中文、SQL 關鍵字、HTML 標籤）

→ 完成後讀取 `_test-plan/write.md`
