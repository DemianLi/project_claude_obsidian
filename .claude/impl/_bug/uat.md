# /bug --uat — UAT 批次回饋分流

## Phase U-1：切分與解析

將甲方的非結構化回饋切分為獨立條目：
- 依編號（1. 2. 3.）、符號（- · *）、換行段落識別邊界
- 每條指派 UAT-{N} 臨時 ID
- 同一段描述涵蓋多個問題時自動拆分

## Phase U-2：自動分類（四型 + 待確認）

| 分類 | 識別信號 | 路由目標 |
|------|----------|----------|
| 🔧 實作缺陷 | 「出現錯誤」「沒有反應」「顯示不正確」——行為與確認需求不符 | `bugs/{BUG-ID}.md` + `bugs-active.md` |
| 📋 需求誤解 | 「我以為應該要…」「當初說的不是這樣」——系統符合需求文件但甲方期望不同 | `_pending/{ID}.md` + `_pending/_index.md` + 澄清問題 |
| 🆕 新需求 | 「希望可以加…」「如果能有…就好了」——需求文件完全未提到 | `/cr` 流程 |
| 🔍 環境問題 | 「只有某台才有」「換瀏覽器就好了」——與程式邏輯無關 | `bugs/{BUG-ID}.md` + `bugs-active.md`（環境標記） |
| ❓ 待確認 | 描述太模糊，無法判斷 | 生成澄清問題 |

> 分類是「推測」，不是判決。U-3 確認前不寫入任何檔案。

## Phase U-3：呈現分流預覽（等待確認）

```
🧪 UAT 分流報告｜{N} 條回饋

━━━ 🔧 實作缺陷（{N} 條）→ bugs/ ━━━
[UAT-001] {描述}
  關聯需求：REQ-FXXX｜嚴重程度：🔴 高（建議）
  → 確認？ (Y / 調整分類 / 調整嚴重程度)

━━━ 📋 需求誤解（{N} 條）→ _pending/ ━━━
[UAT-002] {描述}
  建議問甲方：「{澄清問題}」
  → 確認？ (Y / 調整分類)

━━━ 🆕 新需求（{N} 條）→ /cr ━━━
[UAT-003] {描述}  → 確認？ (Y / 調整分類)

━━━ ❓ 待確認（{N} 條）━━━
[UAT-004] {描述}
  建議問甲方：「{澄清問題}」

━━━ 操作選擇 ━━━
(A) 全部確認，開始寫入  (B) 逐條確認  (C) 先修改分類
```

## Phase U-4：寫入與路由

確認後依類型分別處理：
- 🔧 實作缺陷 → `bugs/{BUG-ID}.md` + `bugs-active.md`（發現方式填「UAT 甲方回饋」）
- 📋 需求誤解 → `_pending/{ID}.md` + `_pending/_index.md`；對應 `functional/{module}.md` 需求加 ⚠️ UAT 標注
- 🆕 新需求 → 逐條提議執行 /cr（不直接寫入）
- 🔍 環境問題 → `bugs/{BUG-ID}.md` + `bugs-active.md`（根因標記 🔍 環境問題）
- ❓ 待確認 → 彙整澄清問題清單

## Phase U-5：生成甲方回覆草稿

```
感謝您的 UAT 回饋，我們已收到 {N} 條問題，整理如下：

【將修復的問題】（預計 {日期} 前完成）
① {描述} — 已確認為系統問題，修復中

【需要進一步確認的問題】
② {描述} → 想請教：{澄清問題}

【功能需求建議】（需另行評估）
③ {描述} — 超出目前合約範疇，我們會另行評估後回覆

如有任何疑問，歡迎再與我們聯繫。
```

## Phase U-6：完成回報

```
✅ UAT 分流完成｜{N} 條回饋已處理
🔧 實作缺陷 → bugs/：{N} 條（BUG-{起}～BUG-{迄}）
📋 需求誤解 → _pending/：{N} 條
🆕 新需求 → 待執行 /cr：{N} 條
❓ 待確認 → 澄清問題清單：{N} 條
```
append 記錄至 `wiki/log.md`。
