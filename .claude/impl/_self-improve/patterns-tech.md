# Phase 2：提取模式 + 第三方服務踩坑

## 2-A. 提取可複用模式（更新 knowledge/patterns/）

若發現 3 次以上重複使用同一解法，提案新增至 `patterns/index.md`：

```
識別到以下可複用模式：

[1] {模式描述}（在 {N} 個專案中驗證）
    建議新增：
    ### PAT-XXX：{模式名稱}
    **驗證專案**：{專案名}（{N} 次驗證）
    **適用情境**：{什麼時候用}
    **注意事項**：{陷阱或限制}
    → 要建立 PAT-{ID}.md 並登記至 patterns/index.md？ (Y/n)
```

## 2-B. 提取第三方服務踩坑（更新 knowledge/tech-stack/）

格式依 `knowledge/_schemas/tech-stack.md`（新服務先確認分類，決定用精簡格式或第三方服務完整格式）。

對每個識別到的服務，提案建立或更新對應的 `knowledge/tech-stack/{slug}.md`：

```
[1] {服務名稱} — {dev-note 日期}
    內容摘要：{一句話}
    建議更新：
    ・踩過的坑：{坑描述}
    ・有效配置：{若有}
    → 要寫入 tech-stack/{slug}.md（新服務則建立並登記至 index.md）？ (Y/n)
```

寫入標準：
- 坑（負面教訓）：一律建議記錄
- 有效配置（正面發現）：若可複用則記錄，純專案特定的不記錄
- 已有記錄的服務：補充進原檔案，不重複建立條目

同時更新 `index.md` 該服務所在行的使用次數與最後使用日期。

→ 完成後讀取 `_self-improve/lessons.md`
