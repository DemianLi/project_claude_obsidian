# 跨專案分割流程（文件涵蓋多個已知專案）

> 與無主暫存不同：使用者知道是哪些專案，只需要把文件拆分。

---

## S-1：掃描並自動分組

讀完整文件後，將每個片段分為：
- **專案A 專屬**：明確只涉及 A 系統的需求、功能、API
- **專案B 專屬**：明確只涉及 B 系統
- **共用內容**：對多個專案都有價值（關係人、技術環境、領域術語、法規）
- **歸屬不明**：無法從文件本身判斷

## S-2：分割預覽（等待確認）

```
📑 跨專案文件分析｜來源：{文件名稱}

📁 專案 A（{N} 個片段）
  [S-A-1] {描述} → 功能需求 / _pending
📁 專案 B（{N} 個片段）
  [S-B-1] {描述} → 非功能需求
🔗 共用（{N} 個）→ 建議複製到所有相關專案
  [S-C-1] 關係人 → stakeholders.md
  [S-C-2] 技術環境 → existing-system.md
❓ 歸屬不明（{N} 個）→ 需指定
  [S-?-1] {描述}

待確認：
① 分組是否正確？
② 共用內容寫入全部 / 只寫主要專案？
③ 歸屬不明的片段，各指定哪個專案？（或略過）
```

## S-3：逐專案依序執行標準 ingest

確認後，對每個專案**獨立**執行 Phase 2 → 3 → 4：

```
「先整理專案 A：」→ Phase 2 確認 → Phase 3 寫入
「專案 A 完成，現在專案 B：」→ Phase 2 確認 → Phase 3 寫入
```

> 不跨兩個專案同時呈現確認報告，避免混淆。

## S-4：完成回報

```
✅ 跨專案文件整理完成｜來源：{文件名稱}
  專案 A — functional/：{N} 條 / _pending/：{N} 條
  專案 B — functional/：{N} 條 / _pending/：{N} 條
  共用內容 — 已寫入 {N} 個專案的 03-client-context/
```

Append 記錄至 `wiki/log.md`。
