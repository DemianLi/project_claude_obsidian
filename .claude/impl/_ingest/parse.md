# Phase 1：解析（只讀，不寫）

---

## 1-1：確認來源與目標專案

```
使用者指定單一專案 → 繼續 1-2
使用者指定多個專案 → 前往 _ingest/cross-project.md
使用者說「不確定」 → 前往 _ingest/unassigned.md
專案不存在 → 提示：「請先執行 /project-init {名稱}，或要我現在幫你建？」
```

## 1-2：深度解析（已知單一專案）

對每個片段做內容分類：

| 類別 | 判斷標準 | 寫入目標 |
|------|----------|---------|
| 📋 功能需求 | 系統「要做什麼」、使用者操作行為 | `functional.md` |
| ⚙️ 非功能需求 | 效能、安全、相容性等品質指標 | `non-functional.md` |
| 👥 關係人資訊 | 姓名、職稱、聯絡方式、決策權 | `stakeholders.md` |
| 🖥️ 現有系統 | 甲方目前的技術、架構、資料庫、整合點 | `existing-system.md` |
| 🏭 領域背景 | 業務流程、法規、術語定義 | `domain-knowledge.md` |
| 📎 原始背景 | 專案緣起、目標願景 | `00-overview.md` 補充區 |

> 一段文字可能含多種類別，應拆開分別處理。

**需求清晰度標記**（功能 / 非功能適用）：
- 🟢 明確：有具體數字、行為描述、明確輸入輸出
- 🟡 模糊：使用「大概」「類似」「彈性」等詞，或缺少驗收條件
- 🔴 矛盾：與文件其他部分或現有確認需求衝突

## 1-3：矛盾偵測

讀取 `projects/{專案}/01-requirements/functional.md`，比對新內容與現有確認需求：

```
若發現衝突 → 記錄：
  衝突：{REQ-ID}（既有）vs 新文件第 {X} 頁
  既有：{...}  新文件：{...}
  → 等待使用者裁決，不自動覆寫
```

→ 完成後前往 `_ingest/confirm.md`（Phase 2）
