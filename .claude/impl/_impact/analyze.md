# Phase 2：四維度影響範疇分析

對輸入的需求/變更，分四個維度逐一分析：

## 2-1. 資料層影響
```
分析問題：需要新增哪些 table/column？現有結構是否需修改？是否涉及資料遷移？

| 影響 | DB Table | 操作 | 說明 |
|------|----------|------|------|
| 🆕 新增 | {表名} | CREATE | {原因} |
| ✏️ 修改 | {表名} | ALTER | 新增欄位 {欄位名}，影響 {N} 筆現有資料 |
| ⚠️ 遷移 | {表名} | MIGRATE | {資料轉換說明} |
```

## 2-2. API 層影響
```
分析問題：需要新增哪些 endpoint？現有 endpoint 是否需調整？是否有 breaking change？

| 影響 | Endpoint | 操作 | 說明 |
|------|----------|------|------|
| 🆕 新增 | POST /api/xxx | CREATE | {功能說明} |
| ✏️ 修改 | GET /api/xxx | 新增欄位 | response 加入 {欄位}，向後相容 |
| ❗ 破壞性 | PUT /api/xxx | 欄位重命名 | 影響所有 API 呼叫端，需版本化 |
```

## 2-3. 既有需求衝突
```
分析問題：新需求是否與現有確認需求語意衝突？是否影響驗收條件？

| 影響程度 | 需求 ID | 需求名稱 | 衝突/關聯說明 |
|----------|---------|----------|--------------|
| ❗ 衝突 | REQ-F003 | {名稱} | {衝突描述} |
| 🔗 關聯 | REQ-F007 | {名稱} | {關聯說明} |
| ✅ 無影響 | REQ-F001 | {名稱} | 不相關 |
```

## 2-4. 技術決策衝突
```
分析問題：是否與 ADR 有衝突？是否需要推翻或修改技術決策？

| ADR | 決策摘要 | 關係 |
|-----|----------|------|
| ADR-003 | {決策摘要} | {是否衝突，若是說明原因} |
```
