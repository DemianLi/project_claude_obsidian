# Stage 3.5：協作衝突偵測

> 在 BEFORE CODING Stage 3 執行。不依賴 Stage 2 是否找到 REQ——純架構任務（沒有對應 REQ）一樣要檢查元件/API 群組鎖定。
> 前提：`projects/{專案名稱}/collab/` 資料夾存在。

---

## Step 1：讀取 collab/ 資料夾

讀取 `projects/{專案名稱}/collab/*.md`（每位工程師一個檔案），合併所有人的鎖定清單。

若資料夾不存在或為空 → 跳過此 Stage，繼續 Stage 4。

## Step 2：識別目標項目（REQ / 元件 / API 群組，可同時命中多類）

- **REQ**：Stage 2 已識別出的目標 REQ ID（可能多個，也可能沒有）
- **元件**：若任務涉及**修改或新增**（非單純查閱）`02-architecture/system-design/{component}.md`，比對 `arch-index.md` 元件清單與任務關鍵字的交集
- **API 群組**：若任務涉及**修改或新增**（非單純查閱）`02-architecture/api-contracts/{group}.md`，比對 `api-contracts/index.md` 群組清單與任務關鍵字的交集

三類任一命中即進入 Step 3；三類都沒命中 → 跳過此 Stage，繼續 Stage 4。

## Step 3：比對鎖定清單

將 Step 2 識別出的所有目標，與 collab/ 中所有人 `active_work` 的鎖定列比對（類型需一致：目標是「元件」只比對類型為「元件」的鎖定列，REQ 只比對 REQ，以此類推）。

**無衝突** → 繼續 Stage 4。

**有衝突** → 計算每筆衝突鎖定距今天數（今天日期 − 鎖定欄位日期）。超過 14 天標記為「⏰ 可能已過期」——這只是提示，**不自動忽略或刪除**該鎖定（鎖定者可能仍在處理，只是還沒跑 `/save` 更新）。執行 Step 4。

## Step 4：衝突警告 + 要求確認

備注欄填寫規則：超過 14 天的鎖定填「⏰ 已 {N} 天未更新，可能已過期」，14 天以內留空。

```
⚠️ 協作衝突偵測

以下項目目前由其他工程師鎖定：

┌────────┬────────────┬──────────┬──────────┬────────────┐
│ 類型    │ 鎖定項目    │ 鎖定工程師│ 鎖定日期 │ 備注       │
├────────┼────────────┼──────────┼──────────┼────────────┤
│ {類型}  │ {項目}     │ {縮寫}   │ {日期}   │ {note}     │
└────────┴────────────┴──────────┴──────────┴────────────┘

繼續實作可能導致與 {縮寫} 的工作產生衝突。

是否仍要繼續？
  Y — 確認繼續（建議先與 {縮寫} 溝通）
  N — 停止，先與對方確認後再繼續
```

**使用者選 N** → 停止，顯示：
```
🛑 已停止。建議步驟：
  1. 聯繫 {工程師名稱}（{縮寫}），確認 {項目} 的進度
  2. 確認後再繼續；若對方已完成，請他在 /save 時解鎖
```

**使用者選 Y** → 繼續 Stage 4，並在 session 結束 `/save` 時提醒：
```
💡 本次跨越鎖定繼續實作了 {項目}（原鎖定人：{縮寫}），
   建議 /save 時更新自己的 collab/{縮寫}.md 鎖定狀態。
```
