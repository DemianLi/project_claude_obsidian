# /bug 快速記錄流程（含 --list / --close）

## Step 1：收集必要資訊

若使用者只給了簡短描述，一次詢問：
```
① 這個問題在哪個功能發現的？
② 嚴重程度？🔴 高（核心功能無法運作）/ 🟡 中（影響使用）/ 🟢 低（細節）
③ 重現步驟？
④ 預期結果？實際結果？
⑤ 來源：TC 測試案例 / 臨時測試 / 甲方回饋？
```
若使用者已給足資訊（含步驟和預期/實際結果），跳過詢問。

## Step 2：自動分類根因

| 描述特徵 | 根因類型 |
|----------|----------|
| 「系統出現 X，但需求說要 Y」 | 🔧 實作缺陷 |
| 「甲方說 X，但我們做成了 Y」 | 📋 需求誤解（需釐清） |
| 「甲方希望有 X 功能，需求文件裡沒有」 | 🆕 新需求（需要 /cr） |
| 「A 電腦正常，B 電腦出問題」 | 🔍 環境問題 |

分類為「推測」，使用者可修改。

## Step 3：關聯需求

從描述識別相關需求，在 `functional.md` 搜尋對應 REQ-ID。
- 若找到：自動填入「關聯需求」欄位
- 若找不到：「這個問題對應哪條需求？或這是新需求？」

## Step 4：寫入 bugs/{BUG-ID}.md + 更新 bugs-active.md

**主檔**：建立獨立檔案 `06-qa-testing/bugs/{BUG-ID}.md`（一 bug 一檔，零衝突）

```markdown
# {BUG-ID}：{標題}
- **狀態：** 🔴 未修復
- **嚴重度：** {嚴重度}
- **關聯需求：** {REQ-ID}
- **根因類型：** {根因}
- **描述：** {描述}
- **重現步驟：**（逐步填入）
- **預期結果：** {預期}
- **實際結果：** {實際}
- **來源：** {來源}
- **發現日期：** {YYYY-MM-DD}
- **修復方案：** —
- **修復日期：** —
```

**薄索引**：Append 至 `bugs-active.md`（一行一 bug，衝突機率極低）

## Step 5：根因為 🆕 新需求時

```
⚠️ 根因為新需求
「{問題描述}」不在現有確認需求範圍內。
建議執行 /cr [{需求描述}] 建立正式變更請求。
```

---

## /bug --close BUG-{N}

1. 詢問：「修復方案是什麼？」
2. 更新 bugs.md 對應條目：狀態→✅ 已關閉，填入修復方案、修復日期、關閉日期
3. 更新狀態摘要計數
4. 若 test-plan 有對應 TC：「TC-{ID} 對應此 bug，需補充測試驗證修復嗎？」

---

## /bug --list

```
📋 未關閉 Bug｜專案：{名稱}

🔴 高（{N} 個）
  BUG-001 {描述} — 修復中
  BUG-003 {描述} — 未修復（已等 {N} 天）

🟡 中（{N} 個）
  BUG-002 {描述} — 未修復

🟢 低（{N} 個）
  （無）
```
