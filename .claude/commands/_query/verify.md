# /query --verify — 理解驗證模式

## 用途

解決兩個核心風險：
1. **術語誤解**：Claude 對甲方術語的理解，是否和知識庫記錄一致？
2. **需求詮釋偏差**：Claude 對某條需求的解讀，是否符合甲方的原始意圖？

## Step 1：讀取相關知識庫內容

依據 `[主題]`（若有指定）或當前活躍專案，讀取：
- `03-client-context/domain-knowledge.md`
- `01-requirements/functional.md`（相關需求）
- `03-client-context/existing-system.md`（若涉及技術）

## Step 2：輸出理解聲明

Claude 用**自己的話**重述理解，不引用原文：

```
🔍 理解驗證｜{主題}

━━━ 術語理解 ━━━
「{術語}」
  我的理解：{Claude 的詮釋}
  ✅ 這個理解正確嗎？

━━━ 需求詮釋 ━━━
REQ-F{N}：「{需求名稱}」
  我的理解：{Claude 的解讀}
  ✅ 這個理解正確嗎？

━━━ 待確認的模糊點 ━━━
❓「{術語/需求}」：我目前的理解是 {詮釋}，但不確定 {不確定的部分}。
```

## Step 3：處理確認結果

**使用者說「正確」** → 繼續，不更新知識庫

**使用者糾正某條理解** → 立即更新 `domain-knowledge.md` 或 `functional.md`：
```markdown
> ⚠️ 理解更新：{YYYY-MM-DD}
> 舊理解：{錯誤理解}
> 正確理解：{使用者糾正後的內容}
> 來源：/query --verify
```

**使用者澄清了模糊點** → 若是需求，走 `/ingest` 確認流程寫入 `functional.md`

## Step 4：驗證完成回報

```
✅ 理解驗證完成
確認正確：{N} 條 / 已更正：{N} 條 / 新增澄清：{N} 條
建議頻率：每次開始新功能模組前 / /ingest 新文件後 / 需求變更後
```
