# Schema：BUG（缺陷記錄）

> 這是所有指令的唯一格式參考。
> `/bug` 寫入時、`/health-check` 驗證時，均以此為準。
> 權威範例見 `.claude/impl/_bug/quick.md` Step 4。

---

## 完整格式

```markdown
# {BUG-ID}：{標題}
- **狀態：** 🔴 未修復
- **嚴重度：** {嚴重度}
- **關聯需求：** {REQ-ID}
- **根因類型：** {根因}
- **描述：** {描述}
- **重現步驟：**（逐步填入）
- **預期結果：** {預期}
- **實際結果：** {實際}
- **來源：** {來源}
- **發現日期：** {YYYY-MM-DD}
- **修復方案：** —
- **修復日期：** —
```

---

## 必填欄位

| 欄位 | 必填 | 說明 |
|------|------|------|
| 標頭 `# {BUG-ID}：{標題}` | ✅ | ID 格式見下方 |
| `**狀態**` | ✅ | 見下方有效值 |
| `**嚴重度**` | ✅ | 見下方有效值 |
| `**根因類型**` | ✅ | 見下方有效值 |
| `**描述**` | ✅ | 至少 1 句 |
| `**預期結果**` / `**實際結果**` | ✅ | 兩者都須填 |
| `**發現日期**` | ✅ | YYYY-MM-DD 格式 |
| `**關聯需求**` | ⬜ 選填 | 找不到對應 REQ 時可留空，但須在 /bug 流程中被詢問過 |
| `**修復方案**` / `**修復日期**` | ⬜ 條件必填 | 狀態為「✅ 已關閉」時兩者皆不可為「—」 |

---

## 有效狀態值

`🔴 未修復` / `修復中` / `✅ 已關閉`

## 有效嚴重度值

`🔴 高`（核心功能無法運作）/ `🟡 中`（影響使用）/ `🟢 低`（細節）

## 有效根因類型值

`🔧 實作缺陷` / `📋 需求誤解` / `🆕 新需求` / `🔍 環境問題`

---

## ID 規則

- `BUG-{NNN}`：三位數字，從 001 開始，同專案內唯一

---

## 健康檢查規則（供 /health-check 使用）

一個 BUG 被視為「不健康」如果：
- 缺少任何必填欄位
- 狀態、嚴重度或根因類型值不在有效列表中
- 發現日期格式不是 YYYY-MM-DD
- 狀態為「✅ 已關閉」但修復方案或修復日期仍為「—」（資料不一致）
- 標頭 ID 與同目錄其他 BUG 重複
