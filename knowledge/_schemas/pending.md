# Schema：PENDING（待澄清需求）

> 這是所有指令的唯一格式參考。
> `/ingest` 寫入時、`/health-check` 驗證時，均以此為準。
> 權威範例見 `01-requirements/_pending/_index.md`「個別檔案格式」section——本檔是該範例的正式 schema 化。

---

## 完整格式

```markdown
# PENDING-{XXX}：{標題}
**原文**：{甲方文件的原始描述}
**來源**：{文件名稱} 第 {X} 頁
**問題**：{為什麼無法作為需求，例如：缺乏量測指標、有歧義}

**優先級**：🔴 阻塞（影響目前 sprint）/ 🟡 重要（本週應確認）/ 🟢 一般（不急）
**阻塞性**：是 / 否（若「是」，說明：哪條 REQ 或哪個功能模組在等這個澄清）
**預期確認日**：{YYYY-MM-DD / 不確定}
**負責追蹤**：{工程師名稱}

**建議問甲方**：
> {可以直接複製給甲方的問題文字}

**狀態**：🕐 等待回覆 / 🔄 已發問等待中 / ✅ 已解決（移入 functional/{module}.md REQ-FXXX）/ ⛔ 已廢棄
**建立日期**：{YYYY-MM-DD}
**最後更新**：—
```

---

## 必填欄位

| 欄位 | 必填 | 說明 |
|------|------|------|
| 標頭 `# PENDING-{XXX}：{標題}` | ✅ | ID 為三位數字，從 001 開始，同專案內唯一 |
| `**原文**` | ✅ | 須可追溯到具體文件段落 |
| `**來源**` | ✅ | 文件名稱 + 頁碼/位置 |
| `**問題**` | ✅ | 說明為何無法直接作為需求 |
| `**優先級**` | ✅ | 見下方有效值 |
| `**阻塞性**` | ✅ | 是/否；若「是」必須附說明文字，不可空白 |
| `**建議問甲方**` | ✅ | 至少 1 句可直接複製使用的問題 |
| `**狀態**` | ✅ | 見下方有效狀態值 |
| `**建立日期**` | ✅ | YYYY-MM-DD 格式 |
| `**預期確認日**` | ⬜ 選填 | 可填「不確定」 |
| `**負責追蹤**` | ⬜ 選填 | 多工程師時重要 |

---

## 有效狀態值

| 狀態值 | 含義 |
|--------|------|
| `🕐 等待回覆` | 已準備問題，尚未發出 |
| `🔄 已發問等待中` | 已發給甲方，等待答覆 |
| `✅ 已解決` | 甲方已答覆，已移入對應 REQ |
| `⛔ 已廢棄` | 不再需要澄清（例如需求本身已取消） |

## 有效優先級值

`🔴 阻塞` / `🟡 重要` / `🟢 一般`

---

## ID 規則

- `PENDING-{XXX}`：三位數字，從 001 開始，同專案內唯一

---

## 健康檢查規則（供 /health-check 使用）

一個 PENDING 被視為「不健康」如果：
- 缺少任何必填欄位
- 優先級或狀態值不在有效列表中
- 阻塞性填「是」但沒有附說明文字
- 建立日期格式不是 YYYY-MM-DD
- 標頭 ID 與同目錄其他 PENDING 重複
- 標頭 ID 格式不是 `PENDING-{三位數字}`
- 同一檔案含超過一個 `# PENDING-` 標頭（違反一檔一條目，`_index.md` 不適用此項）
