# Schema：REQ（功能需求）

> 這是所有指令的唯一格式參考。
> `/ingest` 寫入時、`/health-check` 驗證時、`/save` 收尾輕量自檢時、`/reconcile` 對帳時，均以此為準。

---

## 完整格式

```markdown
## REQ-F{XXX}：{需求名稱}

**狀態**：{見「有效狀態值」}
**確認日期**：{YYYY-MM-DD}
**來源文件**：{文件名稱} 第 {X} 頁 / 會議記錄 {YYYY-MM-DD} / 圖片輸入（YYYY-MM-DD）
**提出人**：{甲方姓名/角色}（若未知填「不明」）
**合約版本**：{原始合約 v1.0 / CR-{N}}
**清晰度**：{🟢 明確 / 🟡 模糊 / 🔴 矛盾}

### 描述
{需求的完整描述，至少 1 句話}

### 驗收條件
- [ ] {條件一}
- [ ] {條件二（至少 1 條）}
```

---

## 必填欄位

| 欄位 | 必填 | 說明 |
|------|------|------|
| 標頭 `## REQ-F{XXX}：{title}` | ✅ | ID 為三位數字，從 001 開始，同專案內唯一 |
| `**狀態**` | ✅ | 見下方有效狀態值 |
| `**確認日期**` | ✅ | YYYY-MM-DD 格式 |
| `**來源文件**` | ✅ | 需可追溯到具體文件或對話 |
| `**合約版本**` | ✅ | 須宣告是原始合約或 CR 追加 |
| `### 描述` | ✅ | 至少 1 句描述性文字 |
| `### 驗收條件` | ✅ | 至少 1 條可測試的條件 |
| `**提出人**` | ⬜ 選填 | 多關係人時重要 |
| `**清晰度**` | ⬜ 選填 | 若不填，預設為 🟢 明確 |

---

## 有效狀態值

| 狀態值 | 含義 | Claude 行為 |
|--------|------|------------|
| `✅ 已確認甲方` | 甲方書面或 email 確認 | 允許實作 |
| `🕐 待確認` | 未取得甲方明確確認 | **停止**，回報後等待 |
| `🔄 變更中` | 甲方提出修改，尚未定版 | **停止**，等待新版本 |
| `❌ 已廢棄` | 明確不再實作 | **停止**，詢問是否有替代需求 |

---

## ID 規則

- `REQ-F{XXX}`：功能需求（Functional）
- `REQ-NF{XXX}`：非功能需求（Non-Functional），存於 `non-functional.md`
- ID 在同一專案內唯一，不跨專案共用

---

## 選用欄位

```markdown
**實作中**：🔒 {名字縮寫}（YYYY-MM-DD）   ← 多工程師時使用
**原文**：{英文原句}                        ← 英文文件時使用
**相關 ADR**：[[04-decisions/ADR-XXX]]     ← 有對應決策時
**相關教訓**：LESSON-XXX                    ← 有對應踩坑記錄時
```

---

## 版本更新記錄格式

當需求變更時，不刪除原始描述，而是加上變更記錄：

```markdown
> **⚠️ 版本更新**：{YYYY-MM-DD} 依 {文件名/CR-N} 更新
> 舊版本：{舊描述一句話}
> 新版本：見上方描述
```

---

## 健康檢查規則（供 /health-check 使用）

一個 REQ 被視為「不健康」如果：
- 缺少任何必填欄位
- 狀態值不在有效狀態列表中
- 確認日期格式不是 YYYY-MM-DD
- 驗收條件為空（`- [ ]` 至少一條）
- 標頭 ID 與同檔案其他 REQ 重複
