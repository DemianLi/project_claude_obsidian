## 個別檔案格式（建立 `_inferred/{ID}.md` 時使用，依據 `knowledge/_schemas/inferred.md`）

### 推測（INF）

實際寫入格式（標頭即推測內容摘要，不需獨立欄位）：
```markdown
INF-{N}：{業務邏輯描述}
  推測依據：{1-2 行關鍵程式碼引用}
  信心度：🟢 高 / 🟡 中 / 🔴 低
  需驗證：{需要問甲方的問題}
```

選用擴充欄位（`/reverse-engineer` 預設不寫入，缺少不算不健康）：
```markdown
**若推測錯誤，實作後果**：{說明推測錯方向的代價——重工？資料損毀？}
**影響的功能模組**：{哪些功能依賴這條推測，若推測錯則需要修改哪些地方}
**狀態**：待驗證
**更新紀錄**：
— `待驗證` → （驗證後填入）
```

### 邏輯缺口（GAP）

實際寫入格式：
```markdown
GAP-{N}：{缺口描述}
  類型：⚡ 狀態機缺口 / 🗃️ 資料孤島 / 🔀 矛盾邏輯 / 🔘 孤立程式碼 / 🔢 魔術數字
  現象：{程式碼中觀察到的不完整/矛盾情形}
  影響：{擴充哪些功能前必須先釐清}
  建議問甲方：「{具體問題文字}」
```

選用擴充欄位（同上，預設不寫入）：
```markdown
**若強行實作的風險**：{說明跳過釐清可能造成的 bug 或資料問題，量化影響程度}
**狀態**：待釐清
**更新紀錄**：
— `待釐清` → （釐清後填入）
```
