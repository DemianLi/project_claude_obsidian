# 使用場景完整清單 — 覆蓋率排查文件

> 此文件列出軟體服務工程師在真實工作中會遇到的所有場景，
> 並標記每個場景目前的覆蓋狀態，供排查系統盲點使用。
>
> **覆蓋標記：**
> - ✅ 完整覆蓋 — 有明確指令或協議處理此場景
> - 🟡 部分覆蓋 — 可透過現有工具間接處理，但沒有專門設計
> - ❌ 未覆蓋 — 現有系統沒有處理機制

---

## 一、專案啟動階段

### 1.1 接到新案子，從零開始建立知識庫

**場景描述：** 甲方確定合作，提供需求規格書、報價書或初步會議記錄，需要建立這個案子的知識庫。

**觸發方式：** `/project-init`

**系統行為：** 建立完整 7 層資料夾結構（00-overview 至 CHANGELOG），生成各個子文件的初始模板。

**覆蓋狀態：** ✅ 完整覆蓋

---

### 1.2 接到新案子，但還沒確定要承接

**場景描述：** 甲方來詢，需要評估工作量、技術可行性，但還沒確定是否接案。這個階段不應該建立正式專案，但需要暫存評估資訊。

**觸發方式：** 無專用機制

**系統行為：** 無；目前 `/project-init` 直接建立正式專案結構，沒有「評估中/詢價中」狀態。

**覆蓋狀態：** ❌ 未覆蓋

**影響：** 評估資訊要嗎散落在其他地方，要嗎建了一個「假專案」，後續管理混亂。

---

### 1.3 根據過去案子估算新案工時

**場景描述：** 收到類似系統的報價需求，想參考以前做過類似功能花了多久、踩過什麼坑，給出較精確的估算。

**觸發方式：** `/query` 搜尋關鍵字

**系統行為：**
1. `knowledge/patterns/index.md` 每條模式含「參考工時」欄位（含歷史區間與影響因子）
2. `knowledge/lessons-learned/index.md` 每條教訓含「耗時記錄」
3. `05-dev-notes/` session 筆記記錄各需求 REQ-ID 與實際工時
4. `/query [功能關鍵字]` 搜尋並彙整相關記錄，提供可作為估算依據的歷史資料

**覆蓋狀態：** ✅ 完整覆蓋

---

### 1.4 承接舊有系統維護案，沒有任何文件

**場景描述：** 甲方把一個運行中的舊系統交給你維護或擴充，前任工程師沒有留下任何文件，只有程式碼和資料庫。

**觸發方式：** `/reverse-engineer [程式碼/DB schema/API 列表]`

**系統行為：**
1. 分析程式碼或 DB 結構
2. 將確定的結構性事實記錄至 `02-architecture/_legacy-analysis.md`
3. 推測業務邏輯記錄至 `01-requirements/_inferred.md`（附信心度）
4. 識別邏輯缺口記錄至 `01-requirements/_inferred.md`（缺口區）
5. 生成建議問甲方的問題清單

**覆蓋狀態：** ✅ 完整覆蓋

---

### 1.5 中途接手另一個工程師做到一半的案子

**場景描述：** 同事離職或案子轉手，接手時有一些文件、一些程式碼，但不確定哪些做完了、哪些是規劃中的。

**觸發方式：** `/ingest [既有文件]` + `/reverse-engineer [既有程式碼]`

**系統行為：** 兩道指令搭配使用，從既有文件提取需求，從程式碼逆向分析實際完成狀態，但沒有自動比對「計畫中的需求」vs「已實作的程式碼」的差距分析。

**覆蓋狀態：** 🟡 部分覆蓋

**缺口：** 沒有「需求完成度追蹤」機制，無法自動判斷哪些需求已有對應程式碼、哪些還沒實作。

---

## 二、需求管理階段

### 2.1 甲方提供正式需求規格書（SRS/BRD）

**場景描述：** 收到 Word 或 PDF 格式的需求文件，內容較完整，需要解析並存入知識庫。

**觸發方式：** 將文件放入 `_inbox/`，執行 `/ingest`

**系統行為：**
1. Phase 1：解析並分類（功能需求 / 非功能需求 / 背景知識等）
2. Phase 2：呈現解析結果，標記清晰/模糊/衝突項目
3. Phase 3：使用者確認後寫入對應文件

**覆蓋狀態：** ✅ 完整覆蓋

---

### 2.2 甲方提供模糊或語意不清的需求

**場景描述：** 收到的需求書充滿「彈性化」「智能化」「使用友善」等模糊描述，無法直接對應成可實作的規格。

**觸發方式：** `/ingest [文件]`

**系統行為：** 模糊需求被分類為 🟡 狀態，存入 `_pending.md`，附上模糊點描述和建議問甲方的澄清問題。`functional.md` 不寫入任何模糊需求。

**覆蓋狀態：** ✅ 完整覆蓋

---

### 2.3 需求來自會議記錄或訪談逐字稿

**場景描述：** 需求收集是透過與甲方開會，會後整理成會議記錄或訪談逐字稿，而非正式文件。

**觸發方式：** `/ingest [會議記錄文字]`

**系統行為：** `/ingest` 設計上可接受任何文字輸入，會解析並分類其中的需求相關陳述。但會議記錄中夾雜的閒聊、題外話、未決討論需要 Claude 自行過濾。

**覆蓋狀態：** 🟡 部分覆蓋

**缺口：** 沒有明確的「會議記錄模式」——不同於正式文件，會議記錄中常有「甲方A說...但甲方B不同意...最後決定...」的結構，目前 `/ingest` 沒有針對這種對話格式特別處理衝突陳述的機制。

---

### 2.4 需求散落在 email 往返中

**場景描述：** 甲方習慣用 email 溝通，需求以「補充說明」「你那邊方便的話可以...」「對了，另外...」的方式零散分布在多封 email 中。

**觸發方式：** 將 email 文字複製後 `/ingest [email 文字]`，或多次 `/ingest` 累積

**系統行為：** 每封 email 可以單獨 `/ingest`，語意相近的內容在後續 `/reconcile` 時會被標出。但沒有自動關聯同一主題的跨 email 需求片段。

**覆蓋狀態：** 🟡 部分覆蓋

**缺口：** 沒有「email 系列追蹤」機制，無法自動追蹤「這封 email 是在回應哪個需求討論串」。

---

### 2.5 甲方內部兩個人給了相互矛盾的需求

**場景描述：** 甲方業務說「訂單要可以取消」，甲方財務說「確認付款後的訂單不能取消」，兩人都是甲方代表，說法互相衝突。

**觸發方式：** `/ingest` 處理這兩份說法時

**系統行為：** `/ingest` Phase 2 的衝突偵測會標記出 🔴 衝突，要求使用者裁決，不會讓兩條矛盾需求都進入 `functional.md`。

**覆蓋狀態：** ✅ 完整覆蓋

**補充說明：** 每條 REQ 在 `functional.md` 中帶有「提出人」與「來源文件」欄位，`/ingest` 寫入時自動帶入，可追蹤衝突雙方各自的原始陳述來源。

---

### 2.6 需求中途發生變更

**場景描述：** 已確認的需求，甲方在開發到一半時告知要改掉，例如從「單一登入」改成「支援 Google SSO」。

**觸發方式：** `/cr [變更描述]` → `/impact` → `/cr --approve` → `/ingest`

**系統行為：** 先透過 `/cr` 建立變更請求，記錄「原因」「提出人」「影響範疇」；`/impact` 分析架構影響；使用者決定後 `/cr --approve` 核准並觸發 `/ingest`，需求寫入時帶 CR 版本標記。

**覆蓋狀態：** ✅ 完整覆蓋（已補齊）

---

### 2.7 同一份文件跨多個專案都有相關資訊

**場景描述：** 甲方的技術說明文件裡，前半段描述 A 系統的需求，後半段描述 B 系統的需求，兩個系統是你各自負責的不同專案。

**觸發方式：** `/ingest [文件]`，回答「這份文件有 A 和 B 兩個專案的內容」時進入跨專案分割流程

**系統行為：**

Phase S-1：Claude 讀完整文件，將內容分為四類——A 專屬 / B 專屬 / 共用（關係人、技術環境、術語）/ 歸屬不明。

Phase S-2：呈現分割預覽，讓使用者確認分組是否正確、調整錯誤的歸屬、決定共用內容是否同時寫入兩個專案。

Phase S-3：逐專案依序執行標準 ingest 確認流程（先 A 後 B，各自獨立確認，不混在同一張報告裡）。共用內容（如甲方關係人、技術環境、領域術語）在使用者選擇「同時寫入」時，會分別寫入兩個專案的 `03-client-context/`。

Phase S-4：完成後回報各專案分別寫入了幾條需求。

**與「不確定歸屬」的差異：**
- 「不確定/不知道屬於哪個專案」→ `_unassigned/`（舊流程不變）
- 「知道是 A 和 B，但文件混在一起」→ 跨專案分割流程（此情境）

**覆蓋狀態：** ✅ 完整覆蓋（已補齊）

---

## 三、開發前置作業

### 3.1 開始實作一個確認的功能

**場景描述：** 收到「幫我做用戶登入功能」這類開發任務，需要在開始寫程式前確認需求狀態、環境限制、已有決策。

**觸發方式：** 直接給 Claude 開發任務

**系統行為：** BEFORE CODING 協議強制依序讀取：
1. 需求狀態（functional.md — 只允許 ✅ 狀態）
2. 甲方技術環境限制（existing-system.md）
3. 領域術語（domain-knowledge.md）
4. 關係人偏好（stakeholders.md）
5. 系統架構（02-architecture/）
6. 已有技術決策（04-decisions/）
7. 跨專案可用模式（knowledge/patterns/）

**覆蓋狀態：** ✅ 完整覆蓋

---

### 3.2 要實作的功能涉及未驗證的推測

**場景描述：** 想實作某功能，但知識庫中有相關的 INF（推測）或 GAP（邏輯缺口），若推測錯誤會導致實作方向偏差。

**觸發方式：** BEFORE CODING 讀取 `_inferred.md` 時發現相關項目

**系統行為：** 強制停止，展開完整的 INF/GAP 細節、推測依據、信心度、若推測錯誤的後果、建議問甲方的具體問題，並提議整理成 email。

**覆蓋狀態：** ✅ 完整覆蓋

---

### 3.3 想用一個技術方案，但不確定甲方環境相容

**場景描述：** 打算用 Redis 做快取，但不確定甲方的 Windows Server 環境有沒有問題；想用 Docker，不確定甲方能不能跑。

**觸發方式：** `/research [技術方案名稱]`

**系統行為：**
1. 讀取 `existing-system.md` 萃取甲方技術環境約束
2. 加入約束條件到搜索 query（如「Redis Windows Server 2016 相容性」）
3. 搜索並收集 3-5 個候選方案
4. 逐一驗證每個方案與約束輪廓的相容性（✅/❌/🟡）
5. 輸出排名結果

**覆蓋狀態：** ✅ 完整覆蓋

---

### 3.4 需要確認自己對甲方行業術語的理解是否正確

**場景描述：** 甲方文件裡寫著「發票對帳」，在這個行業/這個甲方的系統裡，這個詞具體指什麼操作？跟一般認知是否有差異？

**觸發方式：** `/query --verify [術語]`

**系統行為：** Claude 用自己的話重述對該術語的理解（不是引用原文），使用者確認或糾正；糾正的內容更新至 `domain-knowledge.md`。

**覆蓋狀態：** ✅ 完整覆蓋

---

### 3.5 查詢以前做過類似功能的解法

**場景描述：** 要做權限系統，想看看以前在其他專案有沒有做過，用什麼方案，踩過什麼坑。

**觸發方式：** `/query [功能關鍵字]`

**系統行為：** 搜尋 `knowledge/patterns/index.md`、`knowledge/lessons-learned/index.md`、`knowledge/tech-stack/index.md`，回傳相關記錄與來源。

**覆蓋狀態：** ✅ 完整覆蓋

---

## 四、實作輔助階段

### 4.1 實作過程中發現需求與技術現實有衝突

**場景描述：** 開始做「即時通知」功能，發現甲方的 PHP 5.6 環境不支援 WebSocket，現有需求在這個環境下無法原樣實作。

**觸發方式：** 開發中向 Claude 說明問題

**系統行為：** AFTER TASK 協議提示記錄新技術決策（ADR）；Claude 應回報衝突，提供選項（降級方案、polling、告知甲方升級環境），不自行決定方向。

**覆蓋狀態：** 🟡 部分覆蓋

**缺口：** 沒有「需求可行性驗證」步驟——在 BEFORE CODING 時只驗證需求狀態，不驗證技術可行性。環境不相容的問題需要在實作中途才發現，而非在開工前就預警。

---

### 4.2 實作完後想記錄下今天踩的坑

**觸發方式：** `/save`

**系統行為：** 建立 `05-dev-notes/YYYY-MM-DD-主題.md`，若有可複用模式提示更新 `knowledge/patterns/`，若有教訓提示更新 `knowledge/lessons-learned/`。

**覆蓋狀態：** ✅ 完整覆蓋

---

### 4.3 實作完後要建立技術決策記錄

**場景描述：** 選擇了用 JWT 而非 Session，或決定用軟刪除而非硬刪除，這些選擇的理由需要被記錄，防止未來的自己或接手者再次翻案。

**觸發方式：** AFTER TASK 時 Claude 主動詢問，或 `/save` 時處理

**系統行為：** 在 `04-decisions/` 建立 ADR 文件，記錄情境、考慮過的選項、最終決策與理由、後果。BEFORE CODING 每次都會讀取 ADR，確保不推翻已有決策。

**覆蓋狀態：** ✅ 完整覆蓋

---

### 4.4 開發到一半被問「這個功能當初是為什麼要這樣設計」

**場景描述：** 甲方或同事問起某個設計決定的原因，需要快速回溯。

**觸發方式：** `/query [功能名稱] [決策關鍵字]`

**系統行為：** 搜尋 `04-decisions/` ADR 以及 `05-dev-notes/`，回傳相關記錄。

**覆蓋狀態：** ✅ 完整覆蓋

---

## 五、測試與品質保證

### 5.1 根據需求產生測試情境清單

**場景描述：** 功能開發完後需要寫測試計畫，希望根據 `functional.md` 的需求自動產生對應的測試情境。

**觸發方式：** `/test-plan` 或 `/test-plan REQ-F{N}`

**系統行為：** 自動讀取 `functional.md` 所有已確認需求，為每條需求產生正向/負向/邊界三類測試案例，存入 `06-qa-testing/test-plan-YYYY-MM-DD.md`。

**覆蓋狀態：** ✅ 完整覆蓋（已補齊）

---

### 5.2 確認測試結果對應到哪條需求

**場景描述：** 某個測試失敗，需要快速找出這個測試對應的是哪條功能需求、哪個 ADR 決策，以確定是需求問題還是實作問題。

**觸發方式：** `/bug TC-{ID} [問題描述]`

**系統行為：** `/bug` 指令記錄失敗的測試案例，自動在 `functional.md` 搜尋對應需求（TC ID 帶 REQ-ID 前綴），分類根因（實作缺陷/需求誤解/新需求），若根因為新需求自動提議執行 `/cr`。

**覆蓋狀態：** ✅ 完整覆蓋（已補齊）

---

### 5.3 UAT（使用者驗收測試）期間，甲方回報的問題需要分類

**場景描述：** 甲方在 UAT 期間回傳一批問題，有的是 bug（實作與需求不符）、有的是需求誤解（甲方當初沒說清楚）、有的是新需求（甲方之前沒想到）。

**觸發方式：** `/bug --uat [貼上甲方回饋全文]`

**系統行為：**

Phase U-1：將甲方的非結構化回饋切分為獨立條目，每條指派臨時 ID（UAT-N）。

Phase U-2：對每條自動推測類型：🔧 實作缺陷（系統行為與需求不符）/ 📋 需求誤解（行為符合需求文件但甲方期望不同）/ 🆕 新需求（需求文件沒有）/ 🔍 環境問題（特定環境才出現）/ ❓ 待確認（描述不足）。

Phase U-3：呈現分流預覽，顯示每條的推測分類、關聯需求、嚴重程度建議，讓使用者確認或調整後再寫入。

Phase U-4：依類型路由——缺陷寫入 `bugs.md`、需求誤解寫入 `_pending.md`（同時在 functional.md 加標注）、新需求提議執行 `/cr`、待確認生成澄清問題清單。

Phase U-5：自動生成可直接回覆甲方的訊息草稿，分段列出「將修復的問題」「需要確認的問題」「需另行評估的新功能建議」「確認非程式問題」。

Phase U-6：完成回報——輸出各類型處理數量統計，append 記錄至 `wiki/log.md`。

**覆蓋狀態：** ✅ 完整覆蓋（已補齊）

---

## 六、知識庫維護

### 6.1 甲方確認了一批之前模糊的需求

**場景描述：** 發給甲方的澄清問題得到了回覆，這些回覆可以升格 `_pending.md` 中的模糊需求。

**觸發方式：** `/ingest [甲方回覆]`

### 9.3 知識庫知識轉移給接手的工程師

- P0（第 1 天，約 2 小時）：系統全貌 → 邊界與風險（scope + _pending）→ 甲方環境限制 → 不能推翻的 ADR

**第一週 Checklist：** 生成新工程師可自主追蹤的完成清單。

與 `--handover` 的差異：`--handover` 是文件的完整翻印；`--onboarding` 是以接手者的認知旅程為設計，有優先序和時間估算。

### 12.1 將網路文章/技術資源解析進個人知識庫

**場景描述：** 讀到一篇 Redis 叢集設計的文章、官方 API 文件，或技術部落格，想讓它變成可被 `/query` 查到的個人知識，而非散落在書籤裡。

**觸發方式：** `ingest [URL]`

**系統行為：**
1. Phase 0-F：WebFetch 抓取網頁 → 衍生 slug → 存至 `.raw/articles/{slug}-{日期}.md`
2. 更新 `.raw/.manifest.json` delta 記錄（避免重複解析同一 URL）
3. 若非中文 → 觸發多語言翻譯流程（0-C）
4. 標準 Phase 1-3 解析 → 在 `wiki/sources/` 建立摘要頁（關鍵洞見 / 提取實體 / 提取概念）
5. 自動更新 `wiki/sources/_index.md` 和 `wiki/index.md`

**覆蓋狀態：** ✅ 完整覆蓋

---

### 12.2 跨 session 查詢個人技術概念（非專案相關）

**場景描述：** 想知道「Kafka 和 RabbitMQ 的差異」或「我之前學的 Saga 模式是怎麼用的」，但這不屬於任何甲方專案，是個人的技術知識。

**觸發方式：** `/query [概念/問題]`（預設 Standard 模式）；急用時 `/query quick: [問題]`

**系統行為：** Standard 模式的問題層次路由自動導向：
- `wiki/concepts/_index.md` → 對應概念頁
- `wiki/entities/_index.md` → 對應工具/服務頁
- `wiki/sources/_index.md` + `wiki/questions/_index.md` → 過往研究與歸檔答案

**覆蓋狀態：** ✅ 完整覆蓋

---

### 12.3 好答案自動歸檔，跨 session 複利複用

**場景描述：** 花了 30 分鐘深入研究了某個問題，得到了一個整合多個來源的好答案，不希望這個答案消失在 chat history，下次遇到同問題還要重新研究。

**觸發方式：** `/query` 回答後，Claude 主動詢問是否歸檔

**系統行為：**
- 符合歸檔條件（整合 3+ 來源 / 推導超過 5 分鐘 / 含決策建議）→ 詢問是否寫入 `wiki/questions/`
- 寫入 `wiki/questions/{問題slug}.md`（含 type/answer_quality/related/sources frontmatter）
- Deep 模式強制歸檔（不詢問）
- 更新 `wiki/questions/_index.md` 和 `wiki/index.md`
- 下次同主題查詢時，Standard 模式會先命中 `wiki/questions/` 直接回答

**覆蓋狀態：** ✅ 完整覆蓋

### 12.4 偵測新知識與現有知識矛盾，標記待解決衝突

**場景描述：** 今天讀到一篇文章說「Redis Cluster 不建議超過 1000 個 key slots」，但之前存的筆記說「可以用更多」，兩者矛盾——系統應該標記出來而非靜默覆寫。

**觸發方式：** `/ingest` 任何來源時自動執行

**系統行為：** Phase 3-4 矛盾偵測：
- 掃描 `wiki/concepts/`、`wiki/entities/`、`wiki/sources/` 中語意相關的頁面
- 偵測到衝突時，在**兩個**衝突頁面均加入 `[!contradiction]` callout（附日期、衝突點說明、引用雙方原文）
- 不靜默覆寫任何現有知識
- `/health-check` 可掃描所有未解決的 `[!contradiction]` callout，列出待處理清單

**覆蓋狀態：** ✅ 完整覆蓋

### 12.5 Session 自動化：開新對話自動恢復工作脈絡

**場景描述：** 每次開新 Claude session 都要花幾分鐘重新解釋「我最近在做什麼、上次工作到哪裡、目前有哪些活躍專案」，非常耗時。

**觸發方式：** 自動（每次新 session 開始時）

**系統行為：** `hooks/hooks.json` 定義四個 hook：
- `SessionStart`：執行 `cat wiki/hot.md`，靜默讀取以恢復最近工作脈絡
- `PostCompact`：Context Compaction 後立即重讀 `wiki/hot.md` 以恢復上下文
- `PostToolUse`（Write/Edit 後）：自動 `git add + commit` wiki/ 和 projects/ 的變更
- `Stop`：偵測變更並提示更新 `wiki/hot.md`

**覆蓋狀態：** 🟡 部分覆蓋

**缺口：** `hooks/hooks.json` 已建立，但 Claude Code 的 hooks 功能需手動在設定中啟用後才會生效；目前 SessionStart 讀取仍須透過 CLAUDE.md 的 `[SESSION START]` 指示手動執行（Claude 每次都會讀，效果等同，但不是真正的自動化 hook）。

### 12.6 Wiki 知識層健康度審查

**場景描述：** wiki 知識庫越來越大，想確認沒有孤兒頁面（沒有任何連結指向它）、沒有死連結（[[指向不存在的頁面]]）、沒有遺留未解決的矛盾標記。

**觸發方式：** `/health-check`（工程層 + wiki 層一起掃描）

**系統行為：** `/health-check` report.md 的 wiki 層健康度段落：
- 孤兒頁面掃描：找出無 wikilink 指向的頁面 → 建議補連結
- 死連結掃描：找出 `[[不存在目標]]` → 建議建立 stub 或移除
- 未解決矛盾掃描：找出所有 `[!contradiction]` callout → 列出衝突頁面對
- `--dashboard` 旗標：輸出可貼入 Obsidian 的 Dataview dashboard 區塊

**覆蓋狀態：** ✅ 完整覆蓋

### 12.7 Obsidian 視覺化：graph view、callouts、資料夾色彩

**場景描述：** 想在 Obsidian 中用 Graph View 看知識間的關係網絡；希望 `[!contradiction]`、`[!key-insight]` 等 callout 有視覺區分；希望 wiki/、projects/ 等資料夾有不同顏色方便辨識。

**觸發方式：** 在 Obsidian 中開啟此資料夾為 vault

**系統行為：**
- `.obsidian/snippets/vault-colors.css` 已建立——含資料夾色彩（wiki=藍、entities=紫、sources=綠、questions=黃、projects=橙）、四種 callout 樣式（contradiction=深紅、gap=橙、key-insight=青綠、stale=灰）
- 所有 wiki 頁面均使用 `[[wikilink]]` 格式 → Obsidian graph view 自動建立連接圖
- `knowledge/_schemas/callouts.md` 定義 callout 格式規範

**覆蓋狀態：** 🟡 部分覆蓋

**缺口：** 需要在 Obsidian → 設定 → Appearance → CSS Snippets 中手動啟用 `vault-colors.css`；Obsidian Local REST API MCP（可讓 Claude 直接讀寫 vault）尚未設定，目前 Claude 透過 file tools 直接讀寫檔案（功能等效，但不是 Obsidian-native API）。
