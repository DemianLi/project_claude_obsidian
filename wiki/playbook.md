# 專案管理系統 — 使用場景手冊（Playbook）

> 本手冊記錄 10 個高頻使用場景的完整操作流程。
> 每個場景包含：觸發情境、調用輸入、系統行為、產出結果。
>
> **測試環境**：eshop-sunrise（旭日貿易 B2B 電商平台）
> **最後驗證**：2026-06-27（全部 10 場景通過，已更新為分檔索引架構；測試於臨時沙箱 `projects/_playbook-test/` 執行，完成後已刪除）

---

## 場景索引

| # | 場景名稱 | 觸發指令 | 適用時機 |
|---|---------|---------|---------|
| 1 | 新專案初始化 | `/project-init` | 接下新甲方專案 |

## 場景 1：新專案初始化

### 觸發情境
接到新甲方專案，需要建立完整的知識庫與資料夾結構。

### 調用輸入
```
/project-init eshop-sunrise
```
或直接描述：「幫我初始化一個新專案，客戶是旭日貿易，做 B2B 電商平台」

### 系統行為
1. 建立 7 層專案資料夾（`00-overview` 到 `06-qa-testing`）
2. 初始化所有「薄索引」（⚡ 標記的快速查閱檔）
3. 建立 Quick Context 頭，讓後續每個指令可以快速掃描而不需全文讀取

## 場景 2：解析甲方文件

### 觸發情境
收到甲方 email 或需求文件，內容混合了清晰需求、模糊描述和背景資訊。

### 調用輸入
```
/ingest [貼上甲方 email 全文]
```
或
```
/ingest https://docs.google.com/...
```

### 系統行為
**Phase 1 — 解析**：逐行分析，標記每條為「清晰需求 / 模糊需求 / 背景資訊 / 範疇外」

**Phase 2 — 確認**：列出分類結果，等你確認或修正

```
📋 解析報告
━━━ ✅ 清晰需求（2 條）━━━
[1] VIP 享 9 折        → 寫入 REQ-F001
[2] PO 號必填          → 寫入 REQ-F002

━━━ 🕐 模糊需求（2 條）━━━
[3] 訂單通知           → 問甲方：email 還是 SMS？觸發時機？
[4] 折扣彈性           → 問甲方：VIP 折扣能手動調整嗎？

確認寫入？(Y 確認 / 或指定修改)
```

**Phase 3 — 寫入**：
- 清晰需求 → 先依關鍵字比對 `functional-index.md` 找模組，寫入對應 `functional/{module}.md`（狀態 ✅），無對應模組則新建並登記至 `functional-index.md`
- 模糊需求 → 建立獨立檔案 `_pending/PENDING-{XXX}.md`（附可直接複製給甲方的問題文字），並登記至 `_pending/_index.md`
- 若新內容與現有 ✅ REQ 矛盾 → **攔截，強制走 /cr**

### 產出結果
| 類型 | 寫入位置 | 狀態標記 |
|------|---------|---------|
| 清晰需求 | `functional/M0X-*.md`（+ 登記 `functional-index.md`） | ✅ 已確認甲方 |
| 模糊需求 | `_pending/PENDING-{XXX}.md`（+ 登記 `_pending/_index.md`） | 🕐 等待回覆 |
| 與現有 REQ 衝突 | 攔截，不寫入 | → 跳至場景 10 |

## 場景 3：實作前範疇確認

### 觸發情境
準備開始實作某功能，需要確認是否在合約範疇內，以及是否有完整需求。

### 調用輸入
描述你要做的任務，系統在 BEFORE CODING 8 段漏斗自動執行：
```
「我要實作 VIP 折扣套用邏輯」
「我要實作 LINE Pay 金流串接」（測試排除場景）
```

### 系統行為
**Stage 2 — scope.md 掃描**：

```
✅ 「VIP 折扣」→ 在範疇（會員管理模組），繼續 Stage 3
❌ 「LINE Pay」→ 明確排除（scope.md 第二期項目）
   → 停止，建議執行 /cr 若甲方堅持
```

**Stage 3 — functional-index.md 掃描**（通過 Stage 2 後）：
```
「訂單取消」→ 在範疇 ✅，但 functional-index.md 無對應 REQ
→ 停止，建議先 /ingest 確認需求再實作
```

### 產出結果
| 判斷結果 | 系統行為 | 你的下一步 |
|---------|---------|----------|
| 在範疇 + 有 REQ ✅ | 繼續展開實作建議 | 開始實作 |
| 在範疇 + 無 REQ | 停止，提示先 /ingest | 取得甲方確認後 /ingest |
| 範疇外 | 停止，建議 /cr | 走場景 4 |

## 場景 4：登記變更請求

### 觸發情境
甲方（或自己發現）提出超出原始合約範疇的需求，需要正式記錄並評估。

### 調用輸入
```
/cr iOS App 行動端下單功能，CEO 提出，希望 Q3 上線
```

### 系統行為
1. 確認該功能確實不在 scope.md（若已在範疇則提醒用 /ingest）
2. 建立 CR 文件，填入提出人、日期、需求描述
3. 初步列出受影響的現有需求
4. 標記複雜度，提示執行 /impact

### 產出結果
`04-decisions/CR-001-ios-app.md`：

```markdown
---
cr-id: CR-001
status: 🕐 評估中
---
# CR-001：iOS App 行動端下單

提出人：陳總（CEO）
需求：VIP 業務可在 iPhone 上直接下單，主要功能：瀏覽商品、加入購物車、送出訂單

影響的現有需求：
- REQ-F001（VIP 識別）
- REQ-F002（PO 表單 → 手機輸入體驗須重新設計）

預估複雜度：🔴 高（後端 62~82h + App 另計）
```

## 場景 5：需求影響分析

### 觸發情境
CR 核准前，或修改現有功能前，需要了解完整影響範疇與工時估算。

### 調用輸入
```
/impact CR-001
```
或
```
/impact 將 VIP 折扣邏輯從結帳層移至訂單建立層
```

### 系統行為
四個維度同步分析：

**維度 1**：掃描所有 REQ，找出直接/間接受影響項目
**維度 2**：推測資料模型變化（新欄位、新表、migration 風險）
**維度 3**：列出受影響的 API 端點與前端顯示邏輯
**維度 4**：工時估算（各子任務 + 30% buffer）

### 產出結果
```
━━━ 影響矩陣 ━━━
REQ-F001  → 直接影響（折扣計算邏輯改變）
REQ-F002  → 間接影響（PO 表單在 App 須重新設計）

━━━ 工時估算 ━━━
後端 API 新增          8h
資料模型調整           2h
VIP 識別整合           4h
前端購物車 & 結帳       16h
QA 回歸測試            8h
Buffer 30%           +11.4h
──────────────────
總計：49.4h（不含 App 本身開發）
```

## 場景 6：推測 / 缺口阻塞

### 觸發情境
實作過程中，BEFORE CODING 漏斗掃描到 `_inferred/_index.md` 有 INF 或 GAP 項目，且與當前任務有交集。

### 調用輸入
```
「我要實作折扣計算 Service」
```
（系統在 Stage 4 自動讀 _inferred/_index.md Quick Context）

### 系統行為
```
Stage 4 — _inferred/_index.md Quick Context 掃描：
  高風險推測：1 個（INF-001 — 阻塞訂單模組）
  GAP 缺口：1 個（GAP-001 — 阻塞折扣計算）← 命中！

→ 讀 _inferred/GAP-001.md 全文
→ 展開 stop-report.md 格式
```

### 產出結果
完整停止報告，含兩個部分：

**Part A — 工程影響說明**：
```
🛑 停止實作｜GAP-001 阻塞折扣計算 Service 架構

問題：同一訂單若同時符合 VIP 折扣 + 業務特定折扣，
      優先序未定義。強行實作將導致架構錯誤，影響所有訂單金額計算。

影響模組：折扣計算 Service、結帳流程、訂單金額欄位
```

**Part B — 可直接複製給甲方的問題文字**：
```
主旨：【旭日電商】折扣優先序確認

林經理您好，

在實作折扣計算模組時，我們發現以下問題需要確認：

當同一筆訂單同時符合「VIP 會員折扣」和「業務人員特定優惠」時：
1. 哪種折扣優先套用？
2. 兩種折扣可以疊加嗎？

例如：VIP 客戶 9 折，業務另給該客戶 85 折，
最終是 9 折？85 折？還是 9 折 × 85 折 = 76.5 折？

此問題影響折扣模組架構設計，確認前暫緩實作。

謝謝
```

## 場景 7：Bug 登記

### 觸發情境
測試或使用中發現功能異常，需要正式記錄並分類。

### 調用輸入
```
/bug VIP 折扣未套用，VIP 帳號結帳時顯示原價而非 9 折價
```

### 系統行為
1. 先讀 `functional-index.md` 找到相關模組，再到對應 `functional/{module}.md` 找出關聯 REQ
2. 分類根因類型（實作缺陷 / 需求不清 / 環境問題 / 設計衝突）
3. 寫入獨立檔案 `bugs/{BUG-ID}.md`（一 bug 一檔，零衝突），同步更新薄索引 `bugs-active.md`

### 產出結果
**`bugs/BUG-001.md`（獨立檔案）**：
```markdown
# BUG-001：VIP 折扣未套用
- 狀態：🔴 未修復
- 嚴重度：🔴 高（核心金流功能，VIP 客戶付全額）
- 關聯需求：REQ-F001（`functional/membership.md`）
- 根因類型：🔧 實作缺陷
- 描述：VIP 結帳時系統顯示原價而非折後金額
- 重現步驟：
  1. 以 VIP 帳號登入
  2. 加入商品至購物車
  3. 進入結帳頁 → 金額顯示原價
- 預期結果：結帳頁顯示 90% 折後金額
- 實際結果：結帳頁顯示 100% 原價
```

**`bugs-active.md`（薄索引同步更新）**：
```
| BUG-001 | VIP 折扣未套用，結帳顯示原價 | 🔴 高 | 未修復 | REQ-F001 |
```

## 場景 8：知識庫查詢

### 觸發情境
想快速了解「某模組現況」「某技術選型原因」「某需求最新狀態」，不想自己翻檔案。

### 調用輸入
```
/query 折扣模組現況如何？
```
或精準模式：
```
/query quick: VIP 折扣相關的所有決策
/query deep: Laravel 軟刪除在這個專案的使用規範
```

### 系統行為
**Standard 模式**（預設）：
1. 掃描 wiki/hot/*.md + wiki/index.md（判斷是否有快取）
2. 若命中 → 從 wiki 層回答
3. 未命中 → 路由至 projects/ 相關檔案掃描
4. 交叉參照所有相關資源（REQ + BUG + GAP + ADR）

### 產出結果
```
📋 折扣模組現況查詢結果

需求（REQ-F001）：VIP 享 9 折 ✅ 已確認
                 → 已由 CR-002 更新為分級折扣制

進行中的 Bug：
  BUG-001：VIP 折扣未套用（🔴 高，未修復）
  → 修復方向已依 CR-002 調整為分級折扣邏輯

阻塞中的缺口：
  GAP-001：多折扣優先序未定義（等甲方回覆）
  → 未解決前折扣計算 Service 暫停實作

建議下一步：追蹤甲方回覆 GAP-001，再繼續 BUG-001 修復
```

若有值得保存的好答案，系統提示歸檔：
```
💡 此答案有複利價值，是否存入 wiki/questions/？(Y/N)
```

## 場景 10：已實作需求被甲方要求修改

### 觸發情境
甲方說「上次確認的需求實作完不對，我要改」——這是最複雜的場景，因為需求已是 ✅ 狀態且已有對應實作。

**範例**：甲方說「VIP 不是固定 9 折，要改成訂單金額分級折扣」

### 調用輸入（三步驟）

**Step 1**（會被攔截）：
```
/ingest [甲方說要改的 email]
```

**Step 2**（系統強制要求）：
```
/cr VIP 折扣改為分級制：<10,000 無折扣 / ≥10,000 9折 / ≥50,000 8折
```

**Step 3**（評估完畢後）：
```
/impact CR-002
/cr --approve CR-002
```

### 系統行為

**Step 1 — 衝突偵測閘（/ingest 攔截）**：
```
❗ 衝突 — 新內容與 REQ-F001（✅ 已確認）發生語意矛盾
  既有：VIP 固定 9 折，無門檻
  新文件：分級折扣，有金額門檻

→ 無法直接寫入，此為實質性需求變更
→ 必須先執行 /cr 建立正式變更請求
```

**Step 2 — /cr 建立**：建立 CR-002，標記「功能修改」類型，列出影響的現有 REQ 與正在修復中的 BUG

**Step 3 — /impact**：分析 DB migration 需求、API 變更、前端文字調整、BUG-001 修復方向改變

**Step 4 — /cr --approve**：
1. 更新 scope.md 追加 CR 記錄
2. 更新 REQ-F001，加入版本標記
3. BUG-001 修復方向同步調整

### 產出結果

**REQ-F001 加版本標記**（`functional/membership.md`）：
```markdown
### REQ-F001：會員分級
合約版本：CR-002

驗收條件：
- VIP，訂單 < NT$10,000  → 無折扣
- VIP，訂單 ≥ NT$10,000  → 9 折
- VIP，訂單 ≥ NT$50,000  → 8 折

> ⚠️ 版本更新（CR-002）：2026-06-23 依甲方確認 email 更新
> 原版本（v1.0）：VIP 結帳自動套用固定 90% 折扣（無金額門檻）
> 🔗 BUG-001 修復方向同步調整為分級折扣邏輯
```

**關鍵機制說明**：

| 機制 | 說明 |
|-----|-----|
| **衝突偵測閘** | /ingest 不允許直接覆寫 ✅ REQ，強制走 /cr |
